# Phase 0.5 — confirmed attribution, corrected mechanism

This document supersedes the [2026-04-24 preliminary attribution](2026-04-24-phase-0.5-attribution-prelim.md). The preliminary's WHERE conclusion (98%+ in `lance_index`) is now confirmed by first-party `phase_timings_ms` instrumentation. The preliminary's WHY conclusion (fragment-count growth) was wrong; the new mechanism hypothesis is version-chain walking, with a real intervention executed but the post-cleanup latency measurement still pending.

| Latency impact verified? | n/a | **No — pending #43 workload retest** |

## TL;DR

- The preliminary attribution at the **phase level** is now confirmed by v2.4.2's `phase_timings_ms` instrumentation. Across 8 individual tail events, `lance_index` accounted for 99.66–99.98% of total `remember()` duration. There is no surprise phase. The work is, definitively, in `LanceStore._index_in_lance()`.
- The **mechanism** I proposed in the preliminary — that fragment count grew with each insert and slowed the manifest — was wrong. Lance's own `get_fragments()` API reports **1 logical fragment** for `notes_cti`. The 7,916 `.lance` files in `data/` were orphan files from uncleaned historical versions (one new version per `INSERT`).
- I executed `cleanup_old_versions()` on 2026-04-25T01:34Z. `notes_cti` shrank from **5.69 GB → 29 MB** (196× reduction); version retained count dropped from "thousands" to **1**.
- **What we don't know yet:** whether the cleanup actually moves the 60-second tail events. The active dataset was already 1 logical fragment, so `compact_files()` was always going to be a no-op. The remaining hypothesis is that LanceDB's insert path walks the version chain (or some metadata structure that scales with history). A workload retest is required to confirm. Tracked as task #43.

## Mechanism correction

### What I claimed (preliminary)

> *Every LanceDB write creates a new fragment. ZettelForge never calls `compact_files()`. As `notes_cti` accumulates fragments, every new insert has to update the table manifest, which is O(fragments).*

### What's actually true

```python
ds = table.to_lance()
fragments = ds.get_fragments()
# len(fragments) == 1
# fragments[0].fragment_id == 7916, fragments[0].count_rows() == 7916
```

Lance's own `get_fragments()` API reports **a single logical fragment** containing all 7,916 rows. The active dataset has always been optimally compacted at the fragment level. `table.compact_files()` correctly returns `fragments_added=0, fragments_removed=0` because there is no fragment merging to do.

The 7,916 `.lance` files in `data/` were **orphans from version history** — every `INSERT INTO ...` cut a new dataset version, leaving the previous version's data files behind. `cleanup_old_versions()` is the API that prunes them.

### Disk impact of the cleanup

```
notes_cti before:  rows=7,916  versions=thousands  files=7,917  size=5.69 GB
notes_cti after:   rows=7,916  versions=1          files=1      size=29 MB
notes_general before: rows=458 versions=468        size=12.7 MB
notes_general after:  rows=458 versions=1          size=3.8 MB
```

`notes_cti` recovered **5.66 GB** (196× shrink). `notes_general` recovered ~9 MB (3.3× shrink). No row loss; `count_rows()` unchanged on both.

### Why this matters for the latency story

Lance's documentation describes `cleanup_old_versions()` as a disk-space hygiene operation, not a hot-path performance fix. But on a workload that creates one new version per insert, the version chain itself is a stack of metadata that the writer may need to walk. Specifically:

- `notes_cti` had thousands of `_versions/*.manifest` entries before cleanup
- Each new write needed to identify the latest manifest, write a new one, and update the version pointer
- LanceDB does not document the cost of "version count" on insert latency, but it is at least a candidate for explaining the 55-second tails

This remains a **hypothesis until task #43 confirms or falsifies it.** If post-cleanup tails persist at ~55 s, then the 5.66 GB of disk-space waste was an unrelated maintenance debt and the real cause of the tail is something we have not yet identified.

## Implications for RFC-009 (revised)

The preliminary's recommendation was "add periodic `compact_files()`." That's wrong — `compact_files()` would have been a no-op all along. Replace with:

**Provisional Phase 1.5 (RFC-009 scope addition, contingent on task #43):**

If the workload retest shows tails collapsing, RFC-009 Phase 1.5 becomes:

- Background thread runs `cleanup_old_versions(older_than=timedelta(hours=1))` on each `<domain>.lance` table on a configurable interval.
- Default cadence: every 60 minutes for write-heavy shards, lower for cold ones.
- Operator override via `lance.cleanup_interval_minutes` config key.
- Telemetry event on every cleanup: bytes freed, versions pruned, elapsed seconds.

If the workload retest shows tails persisting after cleanup, **the diagnosis is not done.** Phase 0.5 returns to active investigation; candidate next causes:

- Disk-IO contention from the 5.66 GB of orphan files (now removed — should resolve on its own)
- An ANN index rebuild that triggers periodically
- Vector quantization step inside `_index_in_lance` that we haven't traced
- Something at the FFI boundary between Python lancedb and the Rust lance crate

The remediation in the preliminary's "Medium-term" section — **moving `_index_in_lance()` into the enrichment outbox** so the vector write is async — remains attractive regardless. It side-steps the entire question by making the user-visible `remember()` no longer wait for the LanceDB write at all. Tracked under RFC-009 §1.1 F03 scope expansion.

## What this document still cannot say

- Whether the post-cleanup `remember()` tail latency is sub-second. **Nothing has run since the cleanup.** Until task #43 produces a histogram of `phase_timings_ms.lance_index` from a representative workload, we have a hypothesis, not a confirmed mechanism.
- Why `notes_general` (458 rows / 468 versions) was fast at ~53 ms while `notes_cti` (7,916 rows / thousands of versions) was slow. The simplest model is that latency scales with version count. The next-simplest model is that it scales with row count. Pre- and post-cleanup workload runs against `notes_cti` distinguish them.
- Whether the 55 ± 4 s clustering of the tail band reflects a specific Lance internal operation. The tightness is a clue (random GC noise wouldn't produce a band that narrow), but we don't yet have a name for it.

## Reproduction

```bash
LOG=/home/rolandpg/.openclaw/workspace-vigil/.zettelforge_vigil/logs/zettelforge.log
SINCE="2026-04-24T23:11:00"

# Aggregate phase share since v2.4.2 went live
jq -s '[.[] | select(.phase_timings_ms != null and .time > "'$SINCE'") | .phase_timings_ms] |
  reduce .[] as $p ({}; reduce ($p | to_entries[]) as $e (.; .[$e.key] = ((.[$e.key] // 0) + $e.value))) |
  (to_entries | map(.value) | add) as $total |
  to_entries | map({phase: .key, total_ms: (.value|round),
                    pct: ((.value * 1000 / $total | round) / 10)}) | sort_by(-.total_ms)' "$LOG"

# Tail events with full phase breakdown
jq -s '[.[] | select(.activity_name=="remember" and .duration_ms > 30000 and .time > "'$SINCE'")] |
  map({time, note_id, duration_ms,
       lance_ms: .phase_timings_ms.lance_index,
       lance_pct: ((.phase_timings_ms.lance_index // 0) * 10000 / .duration_ms | round / 100)})' "$LOG"

# Verify Lance's own fragment count vs disk-file count (the smoking gun)
python3 -c "
import lancedb
t = lancedb.connect('/home/rolandpg/.openclaw/workspace-vigil/.zettelforge_vigil/vectordb').open_table('notes_cti')
print('rows:', t.count_rows())
print('logical fragments:', len(t.to_lance().get_fragments()))
print('versions retained:', len(t.list_versions()))
"
find /home/rolandpg/.openclaw/workspace-vigil/.zettelforge_vigil/vectordb/notes_cti.lance/data -name '*.lance' | wc -l

# How to perform the cleanup (already executed 2026-04-25T01:34Z)
python3 -c "
```
